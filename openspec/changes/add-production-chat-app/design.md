## Context

目标产品是一款面向真实用户的移动端聊天应用，首期以 iOS / Android 双端 Flutter App + NestJS 单体模块化后端为主，要求支持稳定收发消息、弱网恢复、多端登录、媒体附件、推送通知、基础风控与后台治理。当前仓库还没有现成实现，因此本设计需要同时定义产品边界、技术选型、模块划分和上线约束。

这个项目的关键限制不是“功能能不能做出来”，而是“做出来后能否持续演进”。聊天产品天然是高耦合场景，如果一开始把实时、存储、推送、用户态、群组态和客户端 UI 全部糅在一起，后续会非常难维护。因此本设计把“模块边界清晰、职责单一、避免大文件”作为一等约束，而不是后期优化项。

建议的系统轮廓如下：

```text
Flutter App
├── app shell / routing / theme
├── features/auth
├── features/conversation
├── features/chat
├── features/media
├── features/profile
└── shared (network, storage, widgets, logger)

NestJS API
├── auth
├── users
├── conversations
├── messages
├── groups
├── media
├── notifications
├── moderation
├── admin
└── infra (db, redis, queue, observability)
```

推荐目录骨架如下，目标是让新成员在 10 分钟内看懂工程入口与功能边界：


## Goals / Non-Goals

**Goals:**

- 提供一套可落地的聊天产品方案，而不是仅供演示的最小闭环。
- 覆盖单聊、群聊、文本/图片/语音/文件消息、已读未读、推送、离线补偿、多端同步。
- 在客户端与服务端都采用易维护的模块化结构，限制单文件职责，避免后续演进时出现超大文件与跨模块硬耦合。
- 设计可水平扩展的实时链路与存储方案，使后续能够逐步扩展到更高并发。
- 把安全、风控、审计、监控、备份和发布治理纳入首期设计，而不是上线后再补。

**Non-Goals:**

- 首期不做端到端加密；采用传输层 TLS + 服务端存储安全控制即可。
- 首期不做音视频通话、朋友圈/动态、机器人平台、消息搜索高亮等扩展能力。
- 首期不拆分微服务；采用模块化单体优先，待业务量证明需要后再按边界拆分。
- 首期不提供 Web 端与桌面端正式产品，但服务端接口和同步模型要为后续多端扩展保留空间。

## Decisions

### 1. 整体架构采用“模块化单体 + 实时网关 + 队列异步任务”

- 决策：NestJS 先采用模块化单体架构，内部按领域拆分模块；实时通信使用 Socket.IO Gateway；推送、媒体后处理、审计等异步任务通过 BullMQ 承载。
- 原因：对首期团队规模和交付节奏更友好，研发与部署复杂度低，同时能保留模块拆分清晰度。聊天业务的复杂点主要在状态协同，不在于一开始就拆成多服务。
- 备选方案：
  - 微服务：扩展性强，但首期会引入服务发现、链路追踪、分布式事务与多仓协同成本，得不偿失。
  - 纯 REST 轮询：实现简单，但实时体验、功耗与弱网表现较差，不适合聊天主场景。

### 2. 客户端采用 Flutter feature-first 目录结构，内部再做轻量分层

- 决策：Flutter 采用 `features/<domain>` 的 feature-first 组织方式，每个 feature 内再拆为 `presentation / application / domain / data`，共享能力进入 `shared`。
- 原因：比纯 Clean Architecture 更易执行，也比按技术类型平铺目录更利于多人协作；后续新增功能时可以在局部 feature 内扩展，避免一个目录越堆越大。
- 备选方案：
  - 全局按 `pages/services/models` 分类：初期快，但很容易跨功能缠绕。
  - 过重的 DDD/clean 目录：规范强，但对 Flutter 首期交付会增加样板代码。

客户端 feature 内部建议进一步固定为如下结构：

```text
features/chat/
├── presentation/      # 页面、子组件、状态绑定
├── application/       # 用例编排、controller、view model
├── domain/            # 实体、值对象、仓储接口
└── data/              # dto、repository impl、remote/local datasource
```

目录规则：

- `presentation` 不直接写网络请求。
- `data` 不直接依赖 Flutter UI 组件。
- `application` 负责把页面交互转成业务用例，不承担具体序列化细节。
- `shared` 只放跨 feature 复用且边界稳定的能力，禁止把 feature 私有逻辑偷放到 `shared`。

### 3. 明确维护性约束：限制单文件复杂度，按职责拆组件和用例

- 决策：客户端和服务端都将“单文件职责收敛”写入实施约束。
- 具体约束：
  - Flutter 页面文件建议控制在 250 行左右，超过 300 行必须拆分为子 widget、controller、view model 或 section builder。
  - NestJS 的 controller/service/repository/gateway 文件建议控制在 300 行左右，复杂分支超过 3 类职责时必须拆分。
  - DTO、实体、查询对象、事件对象、mapper、常量与校验器不得混写在同一大文件中。
  - 禁止出现“万能 service”“万能 page”“万能 utils”；通用逻辑必须以明确边界沉淀到模块或 shared 层。
- 原因：聊天系统迭代快，功能密度高，不先限制文件膨胀，后期几乎必然出现难以维护的大文件。
- 备选方案：
  - 靠 code review 口头约束：执行力不足，容易失控。
  - 只在后期重构：成本高，风险大。

### 3.1 服务端目录结构固定为“模块内聚，基础设施下沉”

- 决策：NestJS 目录以 `modules/<domain>` 为主，每个模块内部优先拆出 `controller`、`service`、`repository`、`dto`、`entities`、`events`、`gateways` 等子目录，而不是把所有内容堆在模块根目录。
- 推荐结构：

```text
modules/messages/
├── controllers/
├── services/
├── repositories/
├── dto/
├── entities/
├── gateways/
└── events/
```

- 规则：
  - 跨模块访问必须通过公开 service 或事件，不直接绕过边界访问别的模块 repository。
  - 公共基础设施统一放 `infra`，禁止每个业务模块各自复制 logger、cache、config 封装。
  - 管理后台能力与用户侧业务能力分目录分层，避免 controller 混写。
- 原因：聊天服务端模块多、状态多，不固定结构很容易出现“文件找不到、职责看不懂”的问题。

### 4. 服务端数据模型采用 PostgreSQL + Redis + S3 兼容对象存储

- 决策：
  - PostgreSQL 存储用户、会话、消息、群组、设备会话、举报、审计等核心数据。
  - Redis 承载在线状态、Socket 连接映射、幂等去重键、短期未读聚合和发布订阅。
  - S3 兼容对象存储承载图片、语音、文件等大对象。
- 原因：这是聊天系统中成熟、稳定、可维护的组合，关系清晰，扩展路径明确。
- 备选方案：
  - 全部用 MongoDB：消息模型灵活，但复杂关系、审计和后台治理会变重。
  - 全部塞数据库：附件与临时态不合适，成本高。

### 5. 实时协议采用“REST 创建资源 + Socket.IO 推送事件”的混合模式

- 决策：
  - 会话创建、历史拉取、附件上传授权、资料更新等走 REST。
  - 消息送达、已读回执、输入中、会话变更、在线状态等走 Socket.IO。
  - 每条消息包含客户端生成的 `clientMessageId` 和服务端生成的 `serverMessageId`。
- 原因：资源型操作更适合 REST，时效型状态更适合实时推送；混合模式可维护性更高，也更利于排障。
- 备选方案：
  - 全部走 WebSocket：协议耦合高，不利于调试和网关治理。
  - 全部走 HTTP：实时体验差。

### 6. 消息可靠性以“客户端幂等 + 服务端确认 + 顺序号同步”保证

- 决策：
  - 客户端发送消息时先本地落草稿/待发送态，并生成 `clientMessageId`。
  - 服务端在持久化成功后返回 ack，并分配 `conversationSequence`。
  - 客户端断线重连时，按每个会话最后已同步 sequence 增量拉取补偿。
  - 已读状态按“会话已读游标”而不是“逐条写回执”建模，降低写放大。
- 原因：这是移动端弱网聊天场景下兼顾可靠性与实现复杂度的方案。
- 备选方案：
  - 只用服务端时间排序：重复消息与乱序更难收敛。
  - 每条消息独立已读记录：写入量大，群聊成本高。

### 7. 推送与多端同步采用“通知触达”和“状态对账”分离

- 决策：
  - 推送只负责唤醒和提醒，不作为最终消息真值来源。
  - 真正的数据一致性由登录后 REST/Socket 增量同步保证。
  - 同一账号允许多设备在线，但后台可以查看并踢出设备会话。
- 原因：APNs/FCM 不保证实时送达顺序，必须与业务同步模型解耦。
- 备选方案：
  - 把推送当消息主通道：一致性不可控。

### 8. 首期鉴权采用手机号/验证码或邮箱验证码 + JWT + 刷新令牌轮换

- 决策：接入方式预留手机号和邮箱两种身份通道，首期至少落一种；服务端采用短期 access token + 长期 refresh token，刷新时强制轮换并绑定设备。
- 原因：适合移动端产品，也便于后续扩展第三方登录。
- 备选方案：
  - 只做静态 token：安全性和设备治理不足。
  - 一开始集成多个第三方 OAuth：业务价值不一定高，维护成本先上来。

### 9. 媒体消息采用“客户端直传对象存储 + 服务端登记元数据”

- 决策：
  - 客户端先通过服务端获取上传凭证，再直传对象存储。
  - 上传完成后调用确认接口，服务端写入附件元数据，并允许作为消息附件发送。
  - 服务端异步生成缩略图、转码状态和安全扫描结果。
- 原因：避免大文件穿透 API 服务，提高可扩展性。
- 备选方案：
  - 文件统一经过 API 服务转发：实现简单，但带宽与扩展压力大。

### 10. 首期工程治理明确采用统一 lint/test/模块边界检查

- 决策：
  - Flutter 端启用 `flutter_lints`、`dart_code_metrics`、单元测试和 widget 测试。
  - NestJS 端启用 ESLint、类型检查、单元测试、集成测试和 API 契约测试。
  - PR 必须通过静态检查、关键测试和文件规模审查。
- 原因：用户已经明确要求代码易维护，工程守门必须前置。
- 备选方案：
  - 先开发后补规范：最容易导致架构债务积累。

## Risks / Trade-offs

- [实时连接在移动弱网环境下抖动明显] → 通过心跳、指数退避重连、按会话 sequence 增量补偿来降低漏消息风险。
- [群聊写入与未读计数在大群场景成本上升] → 首期限制群规模，采用会话级已读游标与聚合未读模型，必要时再拆分写扩散策略。
- [附件审核或转码耗时导致消息状态复杂] → 将附件状态建模为独立生命周期，消息可先进入“处理中”，客户端展示明确状态。
- [首期不做端到端加密可能不满足部分高安全场景] → 在 proposal 中明确为非目标，后续如面向高敏场景再独立立项。
- [模块化单体在团队扩张后可能出现边界被侵蚀] → 在仓库结构、lint 规则和 code review 模板中强制限制跨模块直接依赖。
- [“文件不要过多”和“文件不要过大”之间存在张力] → 这里的原则不是机械拆文件，而是按职责拆文件；目录先清晰、模块先稳定，再控制单文件大小。

## Migration Plan

1. 先搭建基础工程骨架：客户端 feature-first 目录、服务端模块化目录、数据库 schema、Redis、对象存储、消息网关基础设施。
2. 优先打通账号登录、单聊、消息可靠发送、历史同步与已读状态，形成最小可运营主链路。
3. 在主链路稳定后增加群聊、附件消息、推送通知、多端同步与后台治理能力。
4. 引入可观测性、备份恢复、压测、灰度发布与回滚方案，完成上线前验收。
5. 上线采用 staging 验证、灰度放量、告警观察、必要时快速回滚到上一个稳定版本。

## Open Questions

- 首期身份体系以手机号为主还是邮箱为主，是否需要同时支持两种注册通道？
- 群聊默认上限是 200、500 还是更高？这会直接影响群状态模型与性能预算。
- 语音消息是否要进入首期，如果进入，是否需要转码和波形预览？
- 是否需要内置通讯录导入，如果需要，隐私合规策略要按哪个地区要求设计？
- 后台管理是否由 NestJS 同仓内实现，还是独立一个 Web 控制台项目？
