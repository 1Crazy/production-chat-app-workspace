## Why

项目需要的不是演示性质的聊天 Demo，也不是只够验证想法的 MVP，而是一套可以真正投入使用的聊天应用：用户要能稳定登录、多端在线、可靠收发消息、接收推送、在弱网下恢复同步，并具备基础治理与运维能力。现在先把产品边界、架构约束和交付路径写清楚，可以避免后续在客户端体验、实时链路、存储模型、安全策略和上线运维上反复返工。

## What Changes

- 定义基于 Flutter + Dart 的移动端应用方案，覆盖注册登录、会话列表、单聊、群聊、媒体消息、联系人发现、个人资料、通知处理、设备会话管理等能力。
- 定义基于 NestJS 的后端方案，覆盖 REST API、WebSocket 实时连接、消息持久化、未读状态、群成员管理、文件元数据、推送分发、后台管理接口等能力。
- 定义可靠消息链路，包括客户端消息 ID、服务端确认、重发补偿、顺序控制、送达回执、已读回执、输入中状态和离线追平。
- 定义图片、语音、文件等附件消息的上传、校验、存储、预览与生命周期管理。
- 定义生产可用所需的安全与治理能力，包括 JWT 鉴权、刷新令牌轮换、设备登录管理、限流、防滥用、举报审核钩子、审计日志、监控告警与备份要求。
- 定义工程与交付约束，重点强调代码可维护性、模块清晰拆分、避免超大文件，以及测试、CI/CD、环境治理和分阶段发布策略。

## Capabilities

### New Capabilities
- `chat-auth-and-identity`: 用户注册登录、令牌刷新、设备会话、用户资料与联系人发现规则。
- `chat-conversations-and-messaging`: 单聊、群聊、消息发送接收生命周期、已读状态、输入中状态、消息历史同步。
- `chat-media-and-files`: 附件上传、校验、对象存储元数据、媒体渲染契约、附件消息生命周期。
- `chat-notifications-and-sync`: 推送通知、角标计数、断线重连补偿、未读计数、多端状态一致性。
- `chat-safety-and-admin`: 举报、限流、审核钩子、后台管理、审计留痕与防滥用控制。
- `chat-platform-operations`: 部署环境、可观测性、可靠性指标、备份恢复、发布治理与工程维护约束。

### Modified Capabilities

- None.

## Impact

- 客户端：新增 Flutter 应用分层、状态管理、本地缓存、推送接入、实时连接、模块化 UI 组织方式。
- 服务端：新增 NestJS 的认证、用户、会话、消息、群组、媒体、通知、审核、后台等模块。
- 数据层：需要关系型数据库承载核心实体，Redis 承载临时态与实时分发能力，对象存储承载媒体文件。
- 接口与协议：需要 REST、WebSocket、附件上传授权协议，以及与推送/审核服务的集成契约。
- 工程与运维：需要 CI/CD、环境配置、密钥管理、日志监控、备份任务、告警与发布回滚机制。
