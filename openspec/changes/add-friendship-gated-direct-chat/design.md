## Context

当前仓库已经具备联系人发现、用户资料页和单聊创建能力，但这三块并没有真实的“好友关系”连接起来。服务端 `resolveTargetUser()` 明确写着“当前还没有好友/邀请体系”，联系人页仍是静态假数据，用户资料页却已经出现“发起单聊”“移除联系人”等只在好友关系存在时才成立的动作。

你给出的原型把这条主链路定义得很清楚：`添加好友`、`好友申请`、`好友列表`、`好友资料页`、`好友可发消息` 是一套连贯状态机；非好友不能直接单聊。因此这次不是简单改一个按钮文案，而是要同时调整服务端权限模型、移动端信息架构和若干页面状态。

## Goals / Non-Goals

**Goals:**

- 建立真实好友关系模型，覆盖好友申请、接受、拒绝、删除和好友列表。
- 让联系人页展示真实好友数据，并对齐原型中的“新的朋友 / 我的好友”结构。
- 把用户资料页动作改成关系状态机：非好友显示添加好友或申请状态，好友显示发消息。
- 将单聊创建权限改成仅允许已接受好友关系的双方创建或复用会话。
- 保持代码可维护：服务端按模块拆分，Flutter 按 feature/domain/data/presentation 拆分，避免继续堆大文件。

**Non-Goals:**

- 本次不实现群聊邀请审批、群好友推荐、朋友圈或复杂社交图谱。
- 本次不引入多级好友分组、标签批量管理或企业通讯录权限模型。
- 本次不实现黑名单与好友关系的联动清算规则以外的复杂安全策略。
- 本次不要求把原型中所有联系人相关页面一次性做满到像素级，只先落核心流程和关键状态。

## Decisions

### 1. 好友申请与好友关系分开建模

- 决策：使用两类持久化对象。
  - `friend_requests`：申请发起、接收、留言、状态、处理时间。
  - `friendships`：双向已建立好友关系，作为联系人页与单聊权限的唯一真值。
- 原因：申请是短生命周期状态流转，好友关系是长期可查询实体，混在一张表里后续会很难维护。
- 备选方案：
  - 单表用状态字段表示申请与好友：查询和唯一约束会越来越混乱。

### 2. 单聊创建权限改为只认已接受好友关系

- 决策：`createOrReuseDirectConversation` 在创建或复用单聊前，必须校验双方已经是 `accepted` 好友；联系人发现只负责暴露资料和触发好友申请，不再作为建聊放行条件。
- 原因：这和你给出的原型一致，也能让“联系人”“新的朋友”“发消息”在产品语义上闭环。
- 备选方案：
  - 保留“可发现即可单聊”：实现简单，但和好友体系冲突，会让联系人页继续失真。

### 3. 用户资料页按关系状态机渲染动作

- 决策：用户资料页至少区分以下状态：
  - `not_friends`: 显示“添加好友”
  - `outgoing_pending`: 显示“等待验证”
  - `incoming_pending`: 显示“接受 / 拒绝”
  - `friends`: 显示“发消息”“删除好友”
- 原因：避免现在这种“既不是好友又能发消息、又能移除联系人”的错误状态。
- 备选方案：
  - 继续用单页静态按钮：实现快，但交互逻辑错误。

### 4. 联系人页拆成真实好友页与新的朋友入口

- 决策：联系人页保留原型里的“新的朋友”快捷入口，但好友列表必须来自服务端真实好友关系；原来的静态分组数据删除。
- 原因：当前联系人页是纯展示壳子，继续往上堆 UI 没意义，应该尽快切回真实数据。
- 备选方案：
  - 暂时保留假数据并新增新页面：会让联系人页真假数据并存，更难维护。

### 5. Flutter 侧按 feature 新增 friendship 模块，不继续挤进 profile/contacts 单文件

- 决策：新增 `features/friendship`，承接好友申请、好友列表、关系状态 DTO/Repository/Controller；`contacts` 保留为联系人入口页，`profile` 只消费关系状态和动作。
- 原因：`profile_page.dart`、`member_profile_page.dart`、`contacts_page.dart` 现在已经偏大，再把好友状态流转继续堆进去会失控。
- 备选方案：
  - 把所有好友逻辑继续放到 `profile` 或 `contacts` 现有文件：短期快，长期难维护。

## Risks / Trade-offs

- [关系状态机分支增多导致前后端容易不一致] → 先在 spec 里固定状态集合与按钮映射，再用测试锁住关键状态。
- [从可发现即可建聊切到必须为好友后，会影响现有直聊入口] → 所有资料页和搜索页入口都先收敛到好友申请，不再混用建聊语义。
- [联系人页从静态页面切到真实数据后，页面改动面较大] → 按 feature 拆分 controller、repository、widgets，避免继续扩大单页文件。
- [好友删除后已有单聊是否还能继续使用存在产品争议] → 本次先在规格里明确，避免实现阶段边做边猜。

## Migration Plan

1. 新增好友申请与好友关系数据模型、仓储和接口。
2. 修改单聊创建权限校验，从发现规则切到好友关系校验。
3. 新增移动端 friendship feature，并让联系人页接入真实好友列表。
4. 修改用户资料页按钮状态机，非好友改为添加好友/申请状态，好友才显示发消息。
5. 回归添加好友、处理申请、好友列表、资料页发消息和非好友禁止单聊等核心流程。

## Open Questions

- 删除好友后是否允许继续访问既有单聊会话，还是应同时禁止继续聊天，这个需要在实现前最终拍板。
