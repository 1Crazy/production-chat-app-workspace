## Context

当前 `chat-auth-and-identity` 的实现只支持验证码注册与验证码登录：移动端登录页明确提示“不使用密码”，服务端 `LoginDto` 与 `AuthService.login()` 也只接受 `identifier + code`。同时，`users` 表还没有密码摘要字段，`verification_codes` 也只按 `identifier` 存储单一验证码记录。

这意味着现有方案更像首期联调链路，而不是长期生产方案。要补齐密码登录，变更会同时影响 Flutter 登录页、Auth DTO/Service、Prisma 模型、验证码用途建模、错误语义、测试与上线兼容策略，属于跨模块且带数据迁移的安全相关改动，需要先把技术边界写清楚。

## Goals / Non-Goals

**Goals:**

- 为新注册账号提供密码设置能力，并将密码作为正式登录凭据。
- 为已有仅验证码登录账号提供“设置密码 / 重置密码”路径，不要求人工迁移。
- 保持现有 access token、refresh token 轮换和设备会话治理模型不变。
- 在安全性、维护性或实现质量明显更优时，允许按工程判断引入成熟依赖来完成密码安全存储、校验与限流扩展。
- 为移动端和服务端定义清晰、可测试的接口与错误语义，便于后续实现与回归测试。

**Non-Goals:**

- 本次不引入第三方 OAuth、MFA、生物识别登录或 SSO。
- 本次不改动联系人发现、资料管理、会话治理等非密码相关能力。
- 本次不要求一次性为所有存量账号批量生成或下发初始密码。
- 本次不新增独立管理端密码治理界面。

## Decisions

### 1. 密码登录作为正式登录路径，验证码保留给注册和密码重置

- 决策：将 `/auth/login` 从验证码登录改为密码登录；验证码继续用于 `/auth/register` 和新增的密码重置流程。
- 原因：登录语义更稳定，减少对验证码下发链路的实时依赖，也更符合用户对常规账号体系的预期。
- 备选方案：
  - 长期同时保留“验证码登录 + 密码登录”：登录分支、风控和 UI 都会变复杂，且验证码登录更容易被滥用。
  - 完全移除验证码：会让注册和密码找回失去身份校验抓手，不可接受。

### 2. 密码摘要方案允许按工程判断选择成熟实现

- 决策：服务端密码摘要实现允许按工程判断选择成熟方案；可以继续使用 Node 内建 `crypto.scrypt`，也可以在安全性、可维护性或团队习惯更优时引入专用密码库，前提是方案边界清晰且不会泄漏明文密码。
- 原因：密码场景对安全与维护性要求高，是否引入依赖应由工程取舍决定，而不是被一条绝对限制卡死。
- 备选方案：
  - 强制只能使用内建能力：在某些场景下可行，但可能错过更成熟的密码库实现。
  - 自行拼接普通散列：安全性不足，不能接受。

### 3. 用户模型增加可空密码字段，存量账号通过“重置密码”完成惰性迁移

- 决策：在 `users` 表增加可空密码摘要与密码更新时间字段；未设置密码的老账号保持 `passwordHash = null`，通过验证码校验后的“设置密码 / 重置密码”接口补齐。
- 原因：避免批量迁移时引入不可控风险，也不需要给现有用户分发一次性初始密码。
- 备选方案：
  - 强制所有老账号重新注册：会破坏现有账号与设备会话连续性。
  - 后台生成默认密码：存在明显安全风险，且用户体验差。

### 4. 验证码记录改为带用途建模，注册与重置密码分开限流和消费

- 决策：给验证码增加 `purpose` 概念，至少区分 `register` 与 `reset-password`；存储与查询按 `identifier + purpose` 处理，请求码接口也显式传递用途。
- 原因：当前验证码只按 `identifier` 存储，注册与重置共用时会互相覆盖，且不利于审计和限流。
- 备选方案：
  - 继续复用单一验证码记录：实现简单，但语义混乱，后续很难扩展更多身份流程。

### 5. 上线采用兼容窗口，先支持新密码能力，再移除旧客户端对验证码登录的依赖

- 决策：分阶段上线。第一阶段先部署支持密码字段、注册设密和重置密码的服务端/客户端；必要时在兼容窗口内保留旧验证码登录分支，仅用于已发布旧客户端平滑过渡。第二阶段在客户端全面升级后移除正式产品面上的验证码登录入口。
- 原因：当前已存在仅会发 `identifier + code` 的客户端实现，直接切断会产生登录失败。
- 备选方案：
  - 一次性硬切：实现最简单，但发布风险最高。

## Risks / Trade-offs

- [密码规则过弱导致撞库风险上升] → 至少要求最小长度、字符复杂度或禁用过弱口令，并对密码登录失败增加独立限流。
- [兼容窗口过长导致双登录模式长期共存] → 明确客户端升级完成后的下线时间点，把验证码登录视为过渡能力而不是长期特性。
- [存量账号未及时设置密码，升级后误以为账号不可用] → 在客户端提供“忘记密码 / 设置密码”入口，并给出“当前账号尚未设置密码”的明确提示。
- [验证码用途改造影响现有请求码逻辑] → 通过接口默认值或短期兼容映射平滑过渡，并补齐接口契约测试。
- [密码摘要字段或日志处理不当带来安全问题] → 严格限制响应序列化、日志脱敏和测试覆盖，避免任何明文泄漏。

## Migration Plan

1. 扩展 Prisma `User` 与 `VerificationCode` 模型，新增密码字段和验证码用途字段，生成迁移脚本。
2. 服务端先落密码哈希、注册设密、密码登录、重置密码接口与兼容逻辑，确保旧数据仍可读。
3. Flutter 登录/注册页切换为密码主流程，并增加“忘记密码 / 设置密码”交互。
4. 对存量账号使用验证码重置流程补齐密码，不做离线批量改密。
5. 回归通过后关闭正式界面的验证码登录入口，并在兼容窗口结束后删除旧登录分支。

## Open Questions

- 兼容窗口是否需要在服务端保留旧验证码登录 API，还是只在预发验证后直接切换到密码登录，这要结合当前移动端发布节奏最终确认。
