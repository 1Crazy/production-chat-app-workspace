# 发布回滚预案

更新时间：2026-04-24

## 触发条件

满足以下任一条件应立即进入回滚评估：

- 发布后出现阻断级登录故障
- 消息发送失败率持续超过 2%，且 10 分钟内无法恢复
- 实时连接全面失效，`chat_realtime_active_connections` 跌到异常低位
- 数据写入异常或关键迁移不可逆失败

## 发布前检查

1. 确认 `pnpm --dir server lint`
2. 确认 `pnpm --dir server typecheck`
3. 确认 `pnpm --dir server test --runInBand`
4. 确认 `cd mobile && flutter analyze`
5. 确认 `cd mobile && flutter test`
6. 记录本次发布版本、上一稳定版本、数据库备份文件名

## 标准发布步骤

1. 先执行发布前数据库备份。
2. 在预发环境按验收清单走完主链路。
3. 生产发布采用小流量或单实例先行。
4. 发布后 15 分钟内重点观察：
   - `/ops/health`
   - `chat_message_delivery_total`
   - `chat_push_delivery_total`
   - `chat_realtime_active_connections`
   - 结构化错误日志

## 回滚步骤

### 应用回滚

1. 锁定到上一稳定镜像或上一稳定构建产物。
2. 回滚应用配置到对应版本。
3. 重启服务并先校验 `/ops/health`。

如果使用当前仓库的 stack 方式部署，建议保留“上一稳定版本”镜像 tag，并按部署环境的 compose 覆盖变量重新拉起。

### 数据回滚

只有当新版本已经写入不兼容数据，且应用代码回滚无法恢复时，才进入数据库回滚。

1. 停止继续写入生产库。
2. 选定最近一次可用备份。
3. 按 [backup-and-restore.md](./backup-and-restore.md) 执行恢复。
4. 恢复后先验证核心链路，再恢复流量。

## 回滚后验收

- 登录成功
- 单聊消息发送成功
- 消息历史分页可用
- 已读同步正常
- 推送唤醒后的增量同步正常
- 管理员封禁 / 踢下线没有异常报错

## 事后复盘要求

- 记录触发时间、触发指标、回滚决策人、回滚完成时间
- 明确是代码问题、配置问题、迁移问题还是外部依赖问题
- 把回滚时实际执行的命令、耗时和卡点补进本文档
